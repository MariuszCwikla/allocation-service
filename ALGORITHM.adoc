= Allocation Algorithm

The core challenge: given a policy with _n_ slots and a pool of available equipment, find the globally optimal assignment. A naive per-slot greedy fails when equipment competes across slots — for example, a monitor that satisfies a hard constraint needed by two different slots.

== Approach: Bipartite Matching

The allocation problem is modelled as a bipartite graph:

* *V-nodes* — policy items (left side, one per required slot)
* *U-nodes* — equipment candidates (right side)
* *Edges* — edge exists between V and U when the candidate meets the policy

The algorithm finds a _maximum cardinality matching_. Allocation succeeds only when every policy slot is matched to a distinct equipment item.

=== Edge Classification

Edges carry a match quality label used for ordering:

[cols="1,3"]
|===
| `FULL_MATCH` | Correct type, condition ≥ minimum, brand = preferred
| `PARTIAL_MATCH` | Correct type, condition ≥ minimum (hard constraints only)
| `NO_MATCH` | Hard constraints not met — no edge drawn
|===

Before matching runs, each V-node's adjacency list is sorted with `FULL_MATCH` edges first. This greedy ordering ensures brand-preferred equipment is tried before generic candidates, increasing the chance of satisfying soft preferences.

=== Main algorithm loop

Algorithm starts with some candidates and runs Bipartite Matching.
If the solution is found, then exit. Otherwise, it fetches next candidates and repeats the algorithm until the solution is found.

In practice solution shall be found in first or in just few iterations.

According to example from the coding task:

> "1 laptop with condition ≥ 0.8, prefer Apple; 2 monitors"

this requires two iterations due to "2 monitors". In first iteration the same monitor will be assigned two two policies.
On second iteration it fetches second monitor and algorithm finishes.

However to limit "infite loops", algorithm applies a hard limit (see link:doc/adr/0002-limit-in-bipartite-matching.adoc[ADR 0002]):

[source]
----
limit = numPolicyItems × 20
----

The repository query orders results by brand match first, then condition score descending, so the solver always sees the best candidates first.

=== Complexity

Bipartite matching via augmenting-path DFS runs in `O(V + E)`. In this problem:

* *V* = policy slots (left) + equipment candidates (right).
Policy slots are capped at 10; candidates per iteration are bounded by `K` — the hard limit
from link:doc/adr/0002-limit-in-bipartite-matching.adoc[ADR 0002], computed as `slots × 20`.
So at most 200 candidates total.
* *E* = edges between a slot and each candidate that passes its hard constraints.
In the worst case every candidate qualifies for every slot, giving `10 × 200 = 2 000` edges.

The outer loop repeats the matching at most `K` times in the absolute worst case, but in practice the solution is found in the first one or two iterations.
Benchmark results on a 5 000-item dataset confirm this — see link:BENCHMARK.adoc[BENCHMARK.adoc].

== Trade-offs vs. Alternatives

See link:doc/adr/0001-matching-algorithm.adoc[ADR 0001] for the full decision record.

[cols="1,2,2", options="header"]
|===
| Algorithm | Pro | Con

| *Bipartite Matching* _(chosen)_
| Simple to implement and test; correct for hard constraints; greedy edge ordering handles soft preferences well in the common case
| Does not _globally_ optimise soft preferences when multiple valid matchings exist

| *Hungarian Algorithm*
| Minimises total cost → globally optimal soft preferences
| Significantly more complex to implement and test

| *Min-cost Flow*
| Most general; handles weighted preferences and capacities
| Overkill for this problem size; substantial added complexity
|===

[[non-optimal-example]]
=== Non-optimal Soft Preference Example

Consider this inventory:

----
Laptop 1  (Apple, score=0.9)
Laptop 2  (Dell,  score=0.9)
Monitor 1 (Dell,  score=0.9)
Monitor 2 (Dell,  score=0.8)
Monitor 3 (Dell,  score=0.85)
----

Policy:

. 1 laptop, prefer Apple
. 1 monitor, score ≥ 0.8
. 1 monitor, score ≥ 0.9

The globally optimal assignment gives Laptop 1 (Apple) to slot 1. The FULL_MATCH-first ordering makes this the likely outcome. However, if the Apple laptop was "stolen" to satisfy another slot via augmentation, the algorithm may settle for Dell — even though a better overall assignment exists. Weighted Bipartite Matching or min-cost flow would guarantee global optimality for soft preferences, at the cost of implementation complexity.

== Candidate Ranking

The `findByPolicyOrdered` repository query ranks candidates as follows:

[source]
----
ORDER BY
  CASE WHEN :brand IS NOT NULL AND e.brand = :brand THEN 0 ELSE 1 END,
  e.conditionScore DESC
----

This favours the preferred brand, then highest condition score. Purchase date (recency) is not yet incorporated.

Simply appending `e.purchaseDate DESC` is not enough — it will order by condition score and then by date.

A better approach: a composite rank score persisted at write time:

[source]
----
rank = daysSinceEpoch(purchaseDate) × W + conditionScore × Y
----

where `W` and `Y` are domain-tuned constants. The column can be indexed, keeping the query fast. This is left for a future iteration.
